# <a name="migrate-from-aspnet-core-22-to-30"></a>從 ASP.NET Core 2.2 遷移至3。0

[Scott Addie](https://github.com/scottaddie)與[Rick Anderson](https://twitter.com/RickAndMSFT)

* 與 ASP.NET Core 2.2 程式碼比較。
* 將相關的變更複製到您的 ASP.NET Core 3.0 專案。

### <a name="framework-reference"></a>架構參考

您可以透過上述其中一個套件取得的 ASP.NET Core 功能，作為 `Microsoft.AspNetCore.App` 共用架構的一部分。 *共用架構* 是安裝在電腦上的一組元件 (*.dll* 檔案) ，其中包含執行時間元件和目標套件。 如需詳細資訊，請參閱[共用的架構](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/) \(英文\)。

* 以 SDK 為目標的專案會 `Microsoft.NET.Sdk.Web` 隱含地參考 `Microsoft.AspNetCore.App` 架構。

  這些專案不需要額外的參考：

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Web">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>
      ...
  </Project>
  ```

* `Microsoft.NET.Sdk`以或 SDK 為目標的專案 `Microsoft.NET.Sdk.Razor` 應該將明確加入 `FrameworkReference` 至 `Microsoft.AspNetCore.App` ：

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Razor">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>

    <ItemGroup>
      <FrameworkReference Include="Microsoft.AspNetCore.App" />
    </ItemGroup>
      ...
  </Project>
  ```

#### <a name="framework-dependent-builds-using-docker"></a>使用 Docker 的 Framework 相依組建

使用相依于 ASP.NET Core [共用架構](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/) 之套件的主控台應用程式的 Framework 相依組建，可能會提供下列執行階段錯誤：

```console
It was not possible to find any compatible framework version
The specified framework 'Microsoft.AspNetCore.App', version '3.0.0' was not found.
  - No frameworks were found.
```

`Microsoft.AspNetCore.App` 是包含 ASP.NET Core 執行時間，而且只存在於 [dotnet/Core/aspnet](https://hub.docker.com/_/microsoft-dotnet-core-aspnet/) Docker 映射上的共用架構。 3.0 SDK 不包含共用架構中所提供的重複程式庫複本，就可以使用 ASP.NET Core 減少 framework 相依組建的大小。 這可能會節省最多 18 MB，但需要 ASP.NET Core 執行時間存在/安裝，才能執行應用程式。

若要判斷應用 (程式是否相依于 ASP.NET 核心共用架構上的直接或間接) ，請檢查在組建/發行應用程式期間產生的檔案 *runtimeconfig.js* 。 下列 JSON 檔案顯示 ASP.NET Core 共用架構的相依性：

```json
{
  "runtimeOptions": {
    "tfm": "netcoreapp3.0",
    "framework": {
      "name": "Microsoft.AspNetCore.App",
      "version": "3.0.0"
    },
    "configProperties": {
      "System.GC.Server": true
    }
  }
}
```

如果您的應用程式使用 Docker，請使用包含 ASP.NET Core 3.0 的基底映射。 例如： `docker pull mcr.microsoft.com/dotnet/core/aspnet:3.0` 。

## <a name="kestrel"></a>Kestrel

### <a name="configuration"></a>組態

將 Kestrel 設定遷移至 (Program.cs) 所提供的[web 主機](#hostb)建立器 `ConfigureWebHostDefaults` ： 

```csharp
public static IHostBuilder CreateHostBuilder(string[] args) =>
    Host.CreateDefaultBuilder(args)
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.ConfigureKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseStartup<Startup>();
        });
```

如果應用程式以手動方式建立主機 `ConfigureWebHost` ，而不是 `ConfigureWebHostDefaults` ，請 `UseKestrel` 在 web 主機建立器上呼叫：

```csharp
public static void Main(string[] args)
{
    var host = new HostBuilder()
        .UseContentRoot(Directory.GetCurrentDirectory())
        .ConfigureWebHost(webBuilder =>
        {
            webBuilder.UseKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseIISIntegration()
            .UseStartup<Startup>();
        })
        .Build();

    host.Run();
}
```

### <a name="kestrel-request-trailer-headers"></a>Kestrel 要求尾端標頭

針對以較早版本的 ASP.NET Core 為目標的應用程式：

* Kestrel 會將 HTTP/1.1 區塊尾端標頭新增至要求標頭集合。
* 要求主體讀取至結尾之後，就可以使用尾端。

這會導致在標頭和結尾之間有一些不明確的問題，因此尾端已移至新的集合 (`RequestTrailerExtensions`) 在3.0 中。

HTTP/2 要求尾端為：

* 在 ASP.NET Core 2.2 中無法使用。
* 以3.0 為形式提供 `RequestTrailerExtensions` 。

有新的要求延伸方法可存取這些尾端。 和 HTTP/1.1 一樣，在要求主體讀取至結尾之後可使用尾端。

在3.0 版本中， `RequestTrailerExtensions` 可以使用下列方法：

* `GetDeclaredTrailers`：取得要求 `Trailer` 標頭，此標頭會列出主體之後所預期的尾端。
* `SupportsTrailers`：指出要求是否支援接收尾端標頭。
* `CheckTrailersAvailable`：檢查要求是否支援結尾和是否可讀取。 這種檢查不會假設有尾端可供讀取。 即使這個方法傳回，也可能不會讀取任何尾端 `true` 。
* `GetTrailer`：從回應取得要求的尾端標頭。 `SupportsTrailers`在呼叫之前檢查 `GetTrailer` ， <xref:System.NotSupportedException> 如果要求不支援尾端標頭，則可能會發生。

如需詳細資訊，請參閱 [將要求尾端放在個別的集合中 (dotnet/AspNetCore #10410) ](https://github.com/dotnet/AspNetCore/pull/10410)。

### <a name="allowsynchronousio-disabled"></a>AllowSynchronousIO 已停用

`AllowSynchronousIO` 啟用或停用同步 i/o Api，例如 `HttpRequest.Body.Read` 、 `HttpResponse.Body.Write` 和 `Stream.Flush` 。 這些 Api 是執行緒耗盡的來源，導致應用程式損毀。 在3.0 中， `AllowSynchronousIO` 預設為停用。 如需詳細資訊，請參閱 [Kestrel 文章中的同步 i/o 一節](../fundamentals/servers/kestrel.md?view=aspnetcore-3.0#synchronous-io)。

如果需要同步 i/o，您可以 `AllowSynchronousIO` 在呼叫時（ `ConfigureKestrel` 例如，如果使用 Kestrel) ）在 (使用的伺服器上設定選項來啟用它。 請注意，伺服器 (Kestrel、HttpSys、TestServer 等 ) 全部都有自己 `AllowSynchronousIO` 的選項，而不會影響到其他伺服器。 您可以使用下列選項，以每個要求為基礎來啟用所有伺服器的同步 i/o `IHttpBodyControlFeature.AllowSynchronousIO` ：

```csharp
var syncIOFeature = HttpContext.Features.Get<IHttpBodyControlFeature>();

if (syncIOFeature != null)
{
    syncIOFeature.AllowSynchronousIO = true;
}
```

如果您對於 <xref:System.IO.TextWriter> 在 [Dispose](/dotnet/standard/garbage-collection/implementing-dispose)中呼叫同步 api 的執行或其他串流有問題，請改為呼叫新的 <xref:System.IO.Stream.DisposeAsync*> api。

如需詳細資訊，請參閱 [所有伺服器中的 [公告] AllowSynchronousIO 已停用 (dotnet/AspNetCore #7644) ](https://github.com/dotnet/AspNetCore/issues/7644)。

### <a name="output-formatter-buffering"></a>輸出格式器緩衝

[Newtonsoft.Json](https://www.newtonsoft.com/json)、 <xref:System.Xml.Serialization.XmlSerializer> 和基礎的輸出格式器 <xref:System.Runtime.Serialization.DataContractSerializer> 只支援同步序列化。 為了讓這些格式器可以使用伺服器的 [AllowSynchronousIO](https://github.com/dotnet/aspnetcore/issues/7644) 限制，MVC 會在寫入磁片之前緩衝處理這些格式器的輸出。 由於緩衝處理的結果，MVC 會在使用這些格式器回應時包含 Content-type 標頭。

<xref:System.Text.Json> 支援非同步序列化，因此格式的格式器不 `System.Text.Json` 會緩衝。 請考慮使用此格式器以改善效能。

若要停用緩衝，應用程式可以 <xref:Microsoft.AspNetCore.Mvc.MvcOptions.SuppressOutputFormatterBuffering> 在啟動時設定：

```csharp
services.AddControllers(options => options.SuppressOutputFormatterBuffering = true)
```

請注意，如果尚未設定，這可能會導致應用程式擲回執行時間例外狀況 `AllowSynchronousIO` 。

### <a name="microsoftaspnetcoreserverkestrelhttps-assembly-removed"></a>已移除 AspNetCore Kestrel. Https 元件

在 ASP.NET Core 2.1 中， *Microsoft.AspNetCore.Server.Kestrel.Https.dll* 的內容已移至 *Microsoft.AspNetCore.Server.Kestrel.Core.dll*。 這是使用屬性的非重大更新 `TypeForwardedTo` 。 若為3.0，則會移除空白 *Microsoft.AspNetCore.Server.Kestrel.Https.dll* 元件和 NuGet 套件。

### <a name="security-middleware-guidance"></a>安全性中介軟體指引

對授權和 CORS 的支援是與 [中介軟體](xref:fundamentals/middleware/index) 方法整合的。 這可讓您在這些案例中使用相同的中介軟體和功能。 此版本提供了更新的授權中介軟體，並增強了 CORS 中介軟體，讓它能夠瞭解 MVC 控制器所使用的屬性。

#### <a name="cors"></a>CORS

先前，CORS 可能很難設定。 已提供中介軟體以用於某些使用案例，但 MVC 篩選準則的用途是在其他使用案例中， **而不** 使用中介軟體。 在 ASP.NET Core 3.0 中，我們建議所有需要 CORS 的應用程式都使用 CORS 中介軟體搭配端點路由。 `UseCors` 可以使用預設原則來提供，而且 `[EnableCors]` 和 `[DisableCors]` 屬性可用來覆寫預設原則（必要時）。

在下例中︰

* 已針對具有命名原則的所有端點啟用 CORS `default` 。
* `MyController`類別會使用屬性來停用 CORS `[DisableCors]` 。

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseCors("default");

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

[DisableCors]
public class MyController : ControllerBase
{
    ...
}
```

在舊版的 ASP.NET Core 中，授權支援是透過屬性提供的 `[Authorize]` 。 未提供授權中介軟體。 在 ASP.NET Core 3.0 中，需要授權中介軟體。 建議您將 ASP.NET Core 授權中介軟體 (`UseAuthorization`) 立即放在之後 `UseAuthentication` 。 授權中介軟體也可以使用可覆寫的預設原則進行設定。

在 ASP.NET Core 3.0 或更新版本中， `UseAuthorization` 會在中呼叫 `Startup.Configure` ，而下列 `HomeController` 需要已登入的使用者：

```csharp

        endpoints.MapDefaultControllerRoute();

public class HomeController : Controller

        ...
    }

使用端點路由時，建議您不要設定 <xref:Microsoft.AspNetCore.Mvc.Authorization.AuthorizeFilter> ，而改為依賴授權中介軟體。  如果應用程式 `AuthorizeFilter` 在 MVC 中使用做為全域篩選，我們建議您重構程式碼，以在的呼叫中提供原則 `AddAuthorization` 。

}
```

有一個新的選項，可控制用戶端的訊息大小限制。 例如，在 `Startup.ConfigureServices` 中：

```csharp
services.AddSignalR(hubOptions =>
{
    hubOptions.MaximumReceiveMessageSize = 32768;
});
```

在 ASP.NET Core 2.2 中，您可以設定 `TransportMaxBufferSize` 和，以有效地控制訊息大小上限。 在 ASP.NET Core 3.0 中，該選項現在只會控制觀察背壓之前的大小上限。

### <a name="mvc-controllers"></a>MVC 控制器

控制器的對應現在會在內部進行 `UseEndpoints` 。

`MapControllers`如果應用程式使用屬性路由，請新增。 因為路由包含 ASP.NET Core 3.0 或更新版本中許多架構的支援，所以新增屬性路由控制器是加入宣告的。

取代下列項目：

* `MapControllerRoute` 的 `MapRoute`
* `MapAreaControllerRoute` 的 `MapAreaRoute`

由於路由現在支援的不只是 MVC，因此術語已經改變，讓這些方法清楚陳述它們的用途。 慣例路由（例如） `MapControllerRoute` / `MapAreaControllerRoute` / `MapDefaultControllerRoute` 會依其新增的順序套用。  (放置更明確的路由，例如區域) 的路由。

在下例中︰

* `MapControllers` 新增屬性路由控制器的支援。
* `MapAreaControllerRoute` 在區域中新增控制器的傳統路由。
* `MapControllerRoute` 新增控制器的一般路由。

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapControllers();
        endpoints.MapAreaControllerRoute(

```

#### <a name="async-suffix-removal-from-controller-action-names"></a>從控制器動作名稱移除非同步尾碼

在 ASP.NET Core 3.0 中，ASP.NET Core MVC 會 `Async` 從控制器動作名稱移除尾碼。 路由和連結產生都會受到這個新的預設值所影響。 例如：

```csharp
public class ProductsController : Controller
{
    public async Task<IActionResult> ListAsync()
    {
        var model = await _dbContext.Products.ToListAsync();
        return View(model);
    }
}
```

在 ASP.NET Core 3.0 之前：

* 您可以在 *Products/metrics.listasync* route 存取上述動作。
* 需要指定尾碼的連結產生 `Async` 。 例如：

    ```cshtml
    <a asp-controller="Products" asp-action="ListAsync">List</a>
    ```

在 ASP.NET Core 3.0 中：

* 您可以在 *Products/List* route 存取上述動作。
* 連結產生不需要指定 `Async` 尾碼。 例如：

    ```cshtml
    <a asp-controller="Products" asp-action="List">List</a>
    ```

此變更不會影響使用屬性指定的名稱 [`[ActionName]`](/dotnet/api/microsoft.aspnetcore.mvc.actionnameattribute) 。 您可以使用中的下列程式碼來停用預設行為 `Startup.ConfigureServices` ：

```csharp
services.AddMvc(options =>
    options.SuppressAsyncSuffixInActionNames = false);
```

#### <a name="changes-to-link-generation"></a>連結產生的變更

如同 [舊版路由的差異](xref:fundamentals/routing#differences-from-earlier-versions-of-routing)檔中所述，使用和類似的 api (連結產生有一些差異 `Url.Link` ，例如) 。 這些包括：

* 根據預設，使用端點路由時，不一定會保留所產生之 Uri 中的路由參數大小寫。 此行為可以使用介面來控制 `IOutboundParameterTransformer` 。
* 產生無效路由的 URI (控制器/動作或不存在的頁面) 將會在端點路由下產生空字串，而不是產生不正確 URI。
* 從目前內容 (路由參數的環境值，不會在與端點路由的連結產生中自動使用) 。 先前，當 (或頁面) 產生另一個動作的連結時，會從 *目前* 的路由環境值推斷未指定的路由值。 使用端點路由時，必須在連結產生期間明確指定所有路由參數。

## <a name="opt-in-to-runtime-compilation"></a>加入執行時間編譯

在 ASP.NET Core 3.0 之前，views 的執行時間編譯是架構的隱含功能。 執行時間編譯會補充視圖的組建階段編譯。 它可讓架構在 Razor 修改檔案時，將視圖和頁面編譯 (的 *cshtml* 檔案) ，而不需要重建整個應用程式。 這項功能支援在 IDE 中進行快速編輯，以及重新整理瀏覽器以查看變更的案例。

在 ASP.NET Core 3.0 中，執行時間編譯是加入宣告的情節。 組建階段編譯是預設啟用的視圖編譯的唯一機制。 在 Visual Studio Code 中，執行時間會依賴 Visual Studio 或 [dotnet 監看式](xref:tutorials/dotnet-watch) ，以在偵測到 *cshtml* 檔案的變更時重建專案。 在 Visual Studio 中，要執行的專案中的 *.cs*、 *cshtml* 或 *Razor* 檔案的變更 (<kbd>Ctrl + F5</kbd>) ，但不是 (<kbd>f5</kbd>) 偵錯工具，會觸發專案的重新編譯。

若要啟用 ASP.NET Core 3.0 專案中的執行時間編譯：

1. 請安裝 [AspNetCore Razor 。>microsoft.aspnetcore.mvc.razor.runtimecompilation](https://www.nuget.org/packages/Microsoft.AspNetCore.Mvc.Razor.RuntimeCompilation) NuGet 套件。
1. 更新 `Startup.ConfigureServices` 以呼叫 `AddRazorRuntimeCompilation` ：

    針對 ASP.NET Core MVC，請使用下列程式碼：

    ```csharp
    services.AddControllersWithViews()
        .AddRazorRuntimeCompilation(...);
    ```
 
    針對 ASP.NET 核心 Razor 頁面，請使用下列程式碼：
 
    ```csharp
    services.AddRazorPages()
        .AddRazorRuntimeCompilation(...);
    ```
 
中的範例 https://github.com/aspnet/samples/tree/master/samples/aspnetcore/mvc/runtimecompilation 顯示在開發環境中有條件地啟用執行時間編譯的範例。

如需檔案編譯的詳細資訊 Razor ，請參閱 <xref:mvc/views/view-compilation> 。

## <a name="migrate-libraries-via-multi-targeting"></a>透過多目標來遷移程式庫

程式庫通常需要支援多個版本的 ASP.NET Core。 大部分針對舊版 ASP.NET Core 編譯的程式庫都應該繼續運作，而不會發生問題。 下列情況需要跨編譯應用程式：

* 程式庫依賴具有二進位 [中斷性變更](#breaking-api-changes)的功能。
* 程式庫想要利用 ASP.NET Core 3.0 中的新功能。 

例如：

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFrameworks>netcoreapp3.0;netstandard2.0</TargetFrameworks>
  </PropertyGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netcoreapp3.0'">
    <FrameworkReference Include="Microsoft.AspNetCore.App" />
  </ItemGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netstandard2.0'">
    <PackageReference Include="Microsoft.AspNetCore" Version="2.1.0" />
  </ItemGroup>
</Project>
```

用 `#ifdefs` 來啟用 ASP.NET Core 3.0 特定 api：

```csharp
var webRootFileProvider =
#if NETCOREAPP3_0
    GetRequiredService<IWebHostEnvironment>().WebRootFileProvider;
#elif NETSTANDARD2_0
    GetRequiredService<IHostingEnvironment>().WebRootFileProvider;
#else
#error unknown target framework
#endif
```

如需在類別庫中使用 ASP.NET 核心 Api 的詳細資訊，請參閱 <xref:fundamentals/target-aspnetcore> 。
